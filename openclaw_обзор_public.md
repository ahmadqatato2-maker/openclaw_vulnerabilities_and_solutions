## Что такое OpenClaw

OpenClaw (ранее ClawdBot, MoltBot) — open-source AI-ассистент (MIT-лицензия), работающий локально и способный:

- Подключаться к мессенджерам: Telegram, Discord, Slack, WhatsApp, Signal, iMessage, Google Chat, Microsoft Teams и др.
- Выполнять shell-команды на хост-системе
- Управлять файлами и каталогами
- **Автоматизировать работу в браузере** (клики, формы, скриншоты, web scraping)
- Интегрироваться с 100+ сервисами через Model Context Protocol (MCP)
- Работать с AI-моделями: Claude (Anthropic), GPT (OpenAI), локальные модели через Ollama

OpenClaw использует архитектуру **«Fat Gateway»** — долгоживущий демон на вашем оборудовании служит центральным узлом, а LLM — статeless-инференс.

**Проект**: GitHub — 60 000+ звёзд, активное развитие по состоянию на февраль 2026.

## Ключевые возможности

### Браузерная автоматизация

OpenClaw имеет встроенные возможности для работы с веб-страницами через **dedicated isolated browser** — изолированный экземпляр Chromium, управляемый через Chrome DevTools Protocol (CDP).

#### Что OpenClaw может делать в браузере

| Действие | Описание |
|---|---|
| **Навигация** | Открывать URL, переходить по ссылкам, вперёд/назад по истории |
| **Клики** | Кликать на кнопки, ссылки, любые интерактивные элементы |
| **Ввод текста** | Заполнять формы, вводить данные в поля |
| **Скроллинг** | Прокручивать страницу вверх/вниз |
| **Drag & drop** | Перетаскивать элементы на странице |
| **Скриншоты** | Делать PNG-скриншоты страницы или отдельного элемента |
| **PDF export** | Сохранять страницу в PDF |
| **Управление вкладками** | Открывать, закрывать, переключаться между вкладками |
| **Чтение контента** | Извлекать текст, атрибуты элементов, структуру DOM |

#### Snapshot System — "зрение" OpenClaw

OpenClaw использует **snapshot-систему** для понимания веб-страниц:

1. **Автоматическая нумерация элементов** — при открытии страницы система анализирует DOM и присваивает номера интерактивным элементам:
   ```
   [1] Кнопка "Войти"
   [2] Поле "Email"
   [3] Поле "Пароль"
   [4] Ссылка "Забыли пароль?"
   ```

2. **Два режима snapshot**:
   - **AI Snapshot** — для сложных сценариев, где AI сам решает, с чем взаимодействовать
   - **Role Snapshot** — для специфичных задач (формы, навигация)

3. **Скриншоты + Vision** — OpenClaw может делать скриншот и отправлять его в vision-модель (Claude Sonnet, GPT-4 Vision) для визуального анализа

**Важно**: OpenClaw не "видит" пиксели как человек — он анализирует структуру DOM и автоматически размечает элементы. Это быстрее и надёжнее визуального распознавания.

#### Архитектура браузерного модуля

```
OpenClaw Gateway
    ↓
Browser Control Service (HTTP API)
    ↓
Chrome DevTools Protocol (CDP)
    ↓
Изолированный Chromium (openclaw profile)
```

#### Режимы работы

| Режим | Описание | Когда использовать |
|---|---|---|
| `openclaw` | Изолированный браузер, управляемый OpenClaw | По умолчанию, для автоматизации |
| `chrome` | Расширение в системном браузере | Когда нужны закладки, сохранённые пароли |

#### Конфигурация

В `~/.openclaw/openclaw.json`:

```json
{
  "browser": {
    "profile": "openclaw",
    "headless": true,
    "port": 18791
  }
}
```

На VPS обычно используется `headless: true` — браузер работает без графического интерфейса.

#### Практические сценарии использования

**Web scraping:**
```
Задача: "Собери цены на товары с сайта X"
OpenClaw:
1. Открывает страницу каталога
2. Делает snapshot (нумерует элементы с ценами)
3. Извлекает текст цен
4. Переходит на следующую страницу
5. Возвращает структурированные данные
```

**Автоматизация форм:**
```
Задача: "Зарегистрируй аккаунт на сервисе Y"
OpenClaw:
1. Открывает страницу регистрации
2. Заполняет email, пароль, имя
3. Проходит проверки (если не CAPTCHA)
4. Кликает "Зарегистрироваться"
```

**Мониторинг изменений:**
```
Задача: "Проверяй раз в час, появилась ли новость на сайте Z"
OpenClaw:
1. Открывает сайт
2. Ищет заголовок с определённым текстом
3. Если найден — отправляет уведомление в Telegram
```

#### Интеграция с Playwright

OpenClaw поддерживает **Playwright MCP** (Model Context Protocol) для продвинутой браузерной автоматизации:
- Работа с authenticated flows (логин, OAuth)
- Управление dynamic UI и client-side state
- Сложные сценарии взаимодействия

#### Ограничения

| Что **не может** | Почему | Обходной путь |
|---|---|---|
| Распознавать CAPTCHA | Anti-bot защита | Сервисы типа 2captcha |
| Работать с Canvas/WebGL "из коробки" | Пиксельная графика, нет DOM | Скриншот + vision-модель |
| Обходить продвинутые anti-bot (Cloudflare Turnstile) | Детекция автоматизации | Дополнительные инструменты |

### Способы взаимодействия с OpenClaw

OpenClaw поддерживает множество способов взаимодействия — от командной строки до мессенджеров. **Telegram не обязателен**, можно работать без него и без SIM-карты.

#### 1. CLI (Command Line Interface)

**Прямая работа через терминал:**

```bash
openclaw chat "Покажи содержимое папки workspace"
openclaw chat
> Привет, что умеешь?
> [агент отвечает]
```

#### 2. Web UI

Доступ через `http://localhost:18790` или `http://<tailscale-ip>:18790`

#### 3. Нативные приложения

| Платформа | Доступность | Способ подключения |
|---|---|---|
| macOS/iOS | Есть | Подключение к gateway через Tailscale |
| Android | Через мессенджеры | Discord, Slack, Telegram и др. |
| Windows | CLI / Web UI | Через браузер или терминал |
| Linux | CLI / Web UI | Через терминал |

#### 4. API вызовы

```bash
curl -X POST http://<tailscale-ip>:18789/api/chat \
  -H "Authorization: Bearer <ваш-токен>" \
  -d '{"message": "Привет, OpenClaw"}'
```

#### 5. Мессенджеры

| Мессенджер | Требует SIM? | Работает на смартфоне? | Бесплатность | Сложность настройки |
|---|---|---|---|---|
| **Telegram** | Нет* | Отлично | Бесплатно | Очень просто |
| **Discord** | Нет | Отлично | Бесплатно | Просто |
| **Slack** | Нет | Отлично | Бесплатный план** | Просто |
| **WhatsApp** | Да | Да | Бесплатно | Средне |
| **Signal** | Да | Да | Бесплатно | Средне |
| **Matrix** | Нет | Да | Бесплатно | Средне |
| **iMessage** | Нет*** | Только iOS | Бесплатно | Только macOS |

\* Для создания Telegram-бота через @BotFather нужен личный Telegram (который может требовать номер), но **сам бот не привязан к номеру**.

\*\* Slack бесплатный план имеет ограничения на историю сообщений (90 дней, 10K сообщений).

\*\*\* iMessage требует Apple ID, но не SIM-карту.

**Можно ли обойтись без мессенджеров?** Да, полностью. OpenClaw работает как standalone-сервис через CLI и Web UI.

## Текущее состояние безопасности

### Критические уязвимости

По данным аудитов 2026 года:

- **CVE-2026-25253** (CVSS 8.8) — remote code execution через WebSocket hijacking на открытых инстансах
- **135 000+ установок** обнаружены открытыми в интернет без защиты (82 страны)
- **12% скиллов на ClawHub** (маркетплейс расширений) содержат вредоносный код
- **40% публичных конфигураций** на GitHub содержат API-ключи в открытом виде
- **Issue #9627** — команды `openclaw update` / `doctor` / `configure` разрешают переменные окружения `${VAR}` и записывают реальные значения в конфиг (патч в версии ≥ 2026.1.29)

### Оценка безопасности

- **Оценка: 2 из 100** — 84% попыток извлечения данных и 91% prompt injection атак успешны
- Кампания «ClawHavoc» — reverse shell в скиллах под видом легитимных инструментов
- Основная проблема: self-hosted архитектура, каждый пользователь отвечает за свою безопасность

## Выбор VPS для развертывания

Рекомендуемые характеристики VPS:

| Вариант | CPU | RAM | Диск | Оценка |
|---|---|---|---|---|
| **Минимальный** | 2 ядра | **4 GB** | 60 GB NVMe | Минимальный рабочий вариант |
| **Оптимальный** | 4 ядра | **4 GB** | 80 GB NVMe | Запас по CPU и диску |
| **С запасом** | 4 ядра | 6 GB | 100 GB NVMe | Для дополнительных сервисов |

**Почему не менее 4 GB RAM:**
- 1-2 GB RAM — не хватит для Docker sandbox в режиме «all» + Node.js + OpenClaw одновременно
- 2 GB RAM — работает впритык, Docker будет использовать swap

### Стратегия масштабирования

Начать с минимального варианта, мониторить нагрузку и при необходимости масштабировать.

**Сигналы для масштабирования:**
- RAM > 85% постоянно (5+ минут)
- CPU load average > 80% (1 min avg)
- Диск > 80% заполнения

## Безопасная установка на VPS

### Этап 1. Подготовка VPS

**1. Обновить систему:**

```bash
sudo apt update && sudo apt upgrade -y
```

**2. Создать выделенного пользователя** (не root):

```bash
sudo useradd -r -m -s /bin/bash openclaw-user
sudo passwd openclaw-user
```

**3. Настроить firewall (UFW):**

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow ssh
sudo ufw enable
```

Порты OpenClaw **не открывать** наружу.

**4. Установить Docker:**

```bash
sudo apt install -y docker.io
sudo usermod -aG docker openclaw-user
```

**5. Установить Node.js 22:**

```bash
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs
```

### Этап 2. Установка OpenClaw

```bash
sudo su - openclaw-user
npm install -g openclaw@latest
```

### Этап 3. Безопасная конфигурация

**1. Привязка к localhost:**

В конфигурации указать `host: 127.0.0.1` (не `0.0.0.0`).

**2. Аутентификация — токен:**

```bash
openssl rand -hex 32
```

Указать токен в конфигурации, ротировать каждые 30 дней.

**3. Права на файлы:**

```bash
chmod 700 ~/.openclaw/
chmod 600 ~/.openclaw/config.yaml
```

**4. API-ключи через переменные окружения:**

```bash
# ~/.openclaw/.env (chmod 600)
ANTHROPIC_API_KEY=sk-ant-api03-...
OPENAI_API_KEY=sk-...
```

В конфиге использовать ссылки:

```json
{
  "models": {
    "providers": {
      "anthropic": {
        "apiKey": "${ANTHROPIC_API_KEY}"
      }
    }
  }
}
```

**Критически важно**:
- API-ключи размещать в корневом `models.providers`, **не** в `skills.entries`
- Не хардкодить ключи напрямую в конфиг (риск Issue #9627)
- Ротировать ключи каждые 3-6 месяцев
- После каждого обновления проверять: `grep -r "sk-ant\|sk-" ~/.openclaw/openclaw.json`

**5. Docker sandbox — режим «all»:**

Все операции (включая основные) выполнять внутри контейнеров.

**6. Политика доступа:**

```yaml
denyByDefault: true
```

Whitelist для конкретных инструментов. Группу `group:runtime` (exec, bash) запретить.

### Этап 4. Удалённый доступ

Gateway OpenClaw по умолчанию слушает на `127.0.0.1:18789` (только localhost). Для удалённого доступа — два варианта.

#### Вариант А: SSH-туннель

```bash
ssh -L 8080:127.0.0.1:18789 openclaw-user@<ваш-vps>
# Теперь localhost:8080 на вашей машине → gateway на VPS
```

**Недостатки:**
- Нужно держать SSH-соединение открытым
- При обрыве — переподключаться вручную
- Неудобно на мобильных устройствах

#### Вариант Б: Tailscale (рекомендуется)

Tailscale — mesh VPN на базе WireGuard. Создаёт приватную сеть между вашими устройствами.

**Установка:**

```bash
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
tailscale ip -4
```

**Конфигурация OpenClaw:**

```json
{
  "gateway": {
    "host": "<TAILSCALE_IP>",
    "port": 18789
  }
}
```

**Преимущества:**
- Всегда доступно (не нужно помнить про SSH-туннель)
- End-to-end шифрование (WireGuard)
- Работает на телефоне, планшете, любых устройствах
- Автоматическое переподключение при смене сети
- Не нужно открывать порты наружу
- Бесплатно до 100 устройств (Personal use)

### Этап 5. После установки

1. **Запустить онбординг**:
   ```bash
   openclaw onboard --install-daemon
   ```

2. **Проверить мониторинг токенов**:
   ```bash
   openclaw status
   openclaw usage
   ```

3. **Не ставить** скиллы с ClawHub без ручной проверки исходного кода

4. **После каждого `openclaw update`** проверять, что ключи не раскрылись:
   ```bash
   grep -r "sk-ant\|sk-" ~/.openclaw/openclaw.json
   ```

5. **Настроить usage limits** в консоли провайдера — защита от утечки ключа

6. Настроить автообновления безопасности ОС:
   ```bash
   sudo apt install -y unattended-upgrades
   sudo dpkg-reconfigure -plow unattended-upgrades
   ```

## Архитектура и доступ

### Схема доступа

```
Вы (ноутбук/телефон)
    │
    ├─→ Telegram/WhatsApp/Discord ──→ Серверы мессенджеров ──→ VPS (gateway :18789)
    │
    ├─→ CLI / Web UI ──→ SSH-туннель/Tailscale ──→ VPS (gateway :18789)
    │
    └─→ macOS/iOS приложение ──→ Tailscale ──→ VPS (gateway :18789)
```

**Два типа подключений:**
- **Мессенджеры** — подключаются через свои серверы; gateway сам поддерживает соединение с API мессенджеров
- **Управляющие клиенты** (CLI, web UI, mobile app) — подключаются к gateway напрямую по WebSocket

### Протокол взаимодействия

**WebSocket** (порт `18789` по умолчанию), не REST. WebSocket выбран из-за событийной природы агента — нужны постоянные двунаправленные каналы для стриминга результатов длительных задач.

**Безопасность протокола:**
- Типизация схем через TypeBox
- Валидация через AJV
- Обязательный handshake на всех соединениях
- По умолчанию gateway слушает только `127.0.0.1:18789`

## Управление сервисом

### Установка демона

OpenClaw устанавливается как **systemd user unit** на Linux:

```bash
openclaw onboard --install-daemon
```

Wizard автоматически:
- Создаёт unit-файл в `~/.config/systemd/user/`
- Включает `loginctl enable-linger` (работа после logout)

### Команды управления

| Действие | Команда |
|---|---|
| Запуск | `clawdbot gateway start` |
| Остановка | `clawdbot gateway stop` |
| Перезапуск | `clawdbot gateway restart` |
| Статус | `clawdbot gateway status` |
| Логи | `clawdbot gateway logs` |

Также работают systemd-команды: `systemctl --user start/stop/restart clawdbot-gateway.service`

**Важно**: gateway должен работать **24/7**. Остановка = все мессенджеры и задачи перестают обрабатываться.

## Техническая схема оплаты и работы с API

### Схема взаимодействия

OpenClaw делает **прямые API-вызовы** к Anthropic/OpenAI, используя **ваш личный API-ключ**. Провайдер видит каждый запрос, считает токены и списывает деньги **напрямую с вашего аккаунта**. OpenClaw не участвует в платежах — это только клиент.

```
Вы → регистрируетесь у AI-провайдера → получаете API-ключ → привязываете оплату
                                                           ↓
                                              Создаёте billing account у провайдера
                                                           ↓
Вы → вводите API-ключ в OpenClaw (на VPS) → OpenClaw делает запросы к API провайдера
                                                           ↓
                               Провайдер видит каждый запрос:
                               - считает токены (input + output + cache)
                               - записывает в ваш billing account
                               - в конце месяца/по порогу списывает с карты
```

**Кто кому платит:**

| Кто | Кому | За что | Как |
|---|---|---|---|
| **Вы** | **AI-провайдер** | Токены LLM | Карта/криптовалюта привязана к billing account |
| **Вы** | **Хостинг** | VPS | Оплата через панель провайдера |
| **OpenClaw** | Никому | — | Open-source, MIT-лицензия |

### Хранение API-ключей

**Расположение**: на вашем VPS, в одном из двух мест.

#### Вариант А: Переменная окружения (рекомендуется)

```bash
# ~/.openclaw/.env (chmod 600)
ANTHROPIC_API_KEY=sk-ant-api03-...
OPENAI_API_KEY=sk-...
```

В конфиге `~/.openclaw/openclaw.json` ссылка на переменную:

```json
{
  "models": {
    "providers": {
      "anthropic": {
        "apiKey": "${ANTHROPIC_API_KEY}"
      }
    }
  }
}
```

**Критически важно**: API-ключи должны быть в корневом уровне `models.providers`, **не** внутри `skills.entries`.

#### Вариант Б: Напрямую в конфиге (небезопасно)

**Проблема**: Issue #9627 — команды `openclaw update` / `doctor` / `configure` могут разрешить `${VAR}` и записать реальные значения в конфиг, раскрыв ключ.

### Как происходит запрос к LLM

1. **Вы пишете сообщение** в Telegram/CLI/Web UI → gateway OpenClaw на VPS
2. **OpenClaw собирает контекст**: инструменты, история, файлы workspace, системный промпт
3. **OpenClaw делает HTTPS-запрос** к API провайдера
4. **Провайдер** проверяет ключ, обрабатывает запрос, считает токены, записывает в usage account, возвращает ответ
5. **OpenClaw** возвращает ответ вам

### Биллинг на стороне провайдера

#### У Anthropic

- **Usage по дням** — сколько токенов использовано
- **Cost breakdown**: Uncached input tokens, Cached input tokens (дешевле на 90%), Output tokens
- **Курсы (Claude Sonnet)**: ~$3 за 1M input tokens, ~$15 за 1M output tokens

#### У OpenAI

- **Usage по дням** — токены, запросы, стоимость
- **По моделям**: GPT-4o, GPT-4, GPT-3.5 — разные цены
- **Курсы (GPT-4o)**: ~$2.50 за 1M input tokens, ~$10 за 1M output tokens

### Контроль расходов

**На стороне провайдера:**
- Установить usage limits / hard limits
- Алерты по email при достижении порога

**На стороне OpenClaw:**

```bash
openclaw status       # Показать токены текущей сессии
openclaw usage        # Включить footer с расходом на каждый ответ
```

### Защита от утечки API-ключа

**Меры защиты:**

1. Ключ хранить в `~/.openclaw/.env` с правами `chmod 600`
2. Никогда не коммитить в git
3. После `openclaw update` проверять, что ключ не раскрылся в конфиге
4. Ротировать ключ каждые 3-6 месяцев
5. Включить usage limits у провайдера (hard limit)
6. Не использовать один ключ на production и testing

**Если ключ утёк:**

1. Немедленно отозвать в консоли провайдера
2. Сгенерировать новый ключ
3. Обновить в OpenClaw (`~/.openclaw/.env`)
4. Проверить billing — нет ли подозрительных запросов
5. Изменить пароль аккаунта у провайдера

### Локальные модели как альтернатива

Можно использовать **Ollama** с локальными моделями (Llama 3, Mistral, DeepSeek и др.):

- API-расходов **нет** ($0)
- Полная приватность — данные не уходят наружу
- Нужен **мощнее VPS**: 8+ GB RAM, желательно GPU
- Качество ниже, чем Claude Sonnet/GPT-4o

**Рекомендация**: гибридный подход — Ollama для простых задач, облачные модели для сложных.

## Мониторинг и поддержка

### A. Постоянный мониторинг

#### A1. Ресурсы VPS

| Метрика | Инструмент | Порог «пора действовать» | Что делать |
|---|---|---|---|
| RAM usage | `free -h`, htop | > 85% постоянно (5+ мин) | Масштабирование или оптимизация Docker |
| CPU load average | `uptime`, htop | > 80% средней (1 min avg) | Масштабирование |
| Диск — занятость | `df -h` | > 80% | Очистка логов/образов Docker или расширение диска |
| Диск — inodes | `df -i` | > 80% | Ротация мелких файлов (логи) |
| Swap usage | `free -h` | Активное использование > 100 MB | Не хватает RAM |
| Сетевой трафик | `vnstat` | Аномальные всплески | Возможная атака или утечка данных |

#### A2. Docker — здоровье контейнеров

| Метрика | Инструмент | На что смотреть |
|---|---|---|
| Статус контейнеров | `docker ps -a` | Контейнеры в статусе `Restarting`, `Exited` |
| Потребление RAM/CPU | `docker stats` | Один контейнер съедает непропорционально много |
| Количество образов | `docker system df` | Накопление неиспользуемых образов |
| Число рестартов | `docker inspect --format='{{.RestartCount}}'` | Рост — memory leak или crash loop |

#### A3. Node.js / OpenClaw

| Метрика | На что смотреть |
|---|---|
| Event loop delay | Задержки > 100 мс — блокировка основного потока |
| Heap memory | Рост без возврата к baseline — утечка памяти |
| GC частота | Учащение GC → приложение под давлением по памяти |
| Время отклика | Деградация latency — ранний сигнал проблем |
| Ошибки в логах | `ERROR`, `FATAL`, stack traces |

#### A4. Безопасность — непрерывный контроль

| Что мониторить | Как | Зачем |
|---|---|---|
| Попытки SSH-входа | `/var/log/auth.log` (fail2ban) | Brute-force атаки |
| Открытые порты | `ss -tlnp` | Ничего кроме SSH не должно слушать на `0.0.0.0` |
| WebSocket-соединения | Логи gateway | CVE-2026-25253 — hijacking |
| Аутентификация gateway | Логи auth-событий | Неавторизованные попытки доступа |
| Целостность конфигурации | `sha256sum` конфигов | Изменение без вашего ведома |
| Ключи в конфиге | `grep -r "sk-ant\|sk-\|bot[0-9]" ~/.openclaw/openclaw.json` | Credential leakage после обновлений |
| Права на `.env` | `ls -l ~/.openclaw/.env` | Должно быть `600` |
| Расходы на API | Консоль провайдера | Необъяснимый рост → утечка ключа |

### B. Мониторинг по событиям

| Событие-триггер | Что проверить сразу |
|---|---|
| **После `openclaw update`** | 1) Ключи не раскрылись в конфиге. 2) Версия ≥ 2026.1.29. 3) Docker sandbox в режиме «all». 4) Права `600` на конфиг и `.env`. 5) API-ключи в `models.providers`, не в `skills.entries` |
| **После установки скилла** | 1) Код проверен вручную. 2) Нет reverse shell, `exec()`, `fetch()` к внешним URL. 3) Не запрашивает `group:runtime` |
| **После перезагрузки VPS** | 1) Docker-демон запущен. 2) Контейнеры в `running`. 3) Firewall активен. 4) Порты не стали слушать на `0.0.0.0` |
| **Подозрительная активность** | 1) Логи — кто отправлял команды. 2) `allowedUsers` не изменился. 3) Нет новых сессий от неизвестных |
| **Ошибки в логах** | 1) Тип ошибки (crash, OOM, auth failure, tool error). 2) `docker logs`. 3) `dmesg` — OOM killer |
| **Получено CVE** | 1) Затрагивает ли вашу версию. 2) Есть ли патч. 3) Временные mitigation-меры |
| **Рост расходов на API** | 1) Логи — объём запросов. 2) Не утёк ли ключ. 3) Нет ли prompt injection |

### C. Мониторинг динамики

| Что отслеживать | Период | Сигнал тревоги |
|---|---|---|
| RAM usage baseline | Неделя → месяц | Постепенный рост → утечка памяти |
| Размер Docker-слоёв | Еженедельно | Постоянный рост → `docker system prune` |
| Размер логов | Еженедельно | Без ротации съедят диск за месяц |
| Число рестартов контейнеров | Неделя | Рост → crash loop |
| Latency ответов | Неделя | Деградация → ресурсы на пределе |
| Расход API-токенов | Неделя → месяц | Необъяснимый рост → prompt injection или утечка |
| Failed SSH-попыток | Неделя | Рост → VPS обнаружен сканерами |
| Swap usage | День | Появление и рост → RAM недостаточно |

### D. Инструменты мониторинга

| Инструмент | Какие пункты закрывает | Сложность |
|---|---|---|
| **htop** | A1 (RAM, CPU, swap) | Минимальная |
| **vnstat** | A1 (сетевой трафик) | Минимальная |
| **docker stats / ps -a** | A2 (все метрики Docker) | Уже есть |
| **docker system df + cron prune** | A2 (образы/слои) | Низкая |
| **fail2ban** | A4 (SSH brute-force) | Низкая |
| **ufw** | A4 (открытые порты) | Минимальная |
| **logrotate** | C (рост логов) | Низкая |
| **journalctl** | A3 (ошибки), A4 (auth) | Уже есть |
| **Cron-скрипт с алертами** | A1, A2, A4, C (пороги) | Средняя |
| **Uptime Kuma** | C (время отклика), A3 (latency) | Средняя |

**Cron-скрипт** — самописный bash, запускается каждые 5 минут. Проверяет пороги и отправляет уведомление в Telegram/email. Заменяет Prometheus + Grafana для небольших VPS.

## Важные вопросы эксплуатации

### Бэкап и восстановление

**Что бэкапить:**
- `~/.openclaw/openclaw.json` — конфигурация
- `~/.openclaw/.env` — API-ключи (хранить зашифрованно!)
- `~/.openclaw/workspace/` — файлы агента, память, промпты
- OAuth-токены и состояние каналов мессенджеров

**Безопасность бэкапов:**
- Шифровать `.env` перед отправкой на удалённое хранилище (GPG)
- Не держать незашифрованные бэкапы с API-ключами на публичных серверах

**Рекомендация**: автоматический бэкап через cron — минимум раз в сутки.

### Обновления

**Риски:**
- Issue #9627 — обновление может перезаписать `${VAR}` в конфиге реальными значениями
- Может сломать совместимость со скиллами
- Может изменить поведение sandbox

**Стратегия**: обновлять вручную, не автоматически. Перед обновлением — снимать снапшот VPS. После — проверять по чеклисту из раздела B.

### Зависимость от внешних сервисов

**Если AI-провайдер упадёт?** Gateway продолжит работать, но агент не сможет генерировать ответы.

**Mitigation:**
- Настроить fallback-модели в конфиге
- Держать локальную модель через Ollama как аварийный вариант
- Для критичных применений — аккаунты у двух провайдеров

### Устойчивость проекта

**Если авторы OpenClaw забросят проект?**

- Код открыт (MIT) — можно форкнуть и поддерживать
- npm-пакет можно зафиксировать на конкретной версии
- Установленные скиллы продолжат работать
- Критичная зависимость — npm registry (иметь локальную копию пакета)

### Мультиинстанс

OpenClaw поддерживает Multiple Gateways — каждый gateway это отдельный systemd unit, отдельный порт, отдельная конфигурация.

### Сессии мессенджеров

- **Telegram**: бот-сессия привязана к токену, переживает перезапуск
- **WhatsApp**: сессия хранится в `~/.openclaw/`, переживает перезапуск, но при миграции иногда требует повторной привязки QR-кодом

### Утечка API-ключа

**Признаки утечки:**
- Необъяснимый рост расходов на API
- Запросы в часы, когда вы не работали
- Непривычные паттерны использования

**Действия при обнаружении:**

1. Немедленно отозвать ключ в консоли провайдера
2. Сгенерировать новый ключ
3. Обновить в OpenClaw
4. Проверить billing
5. Включить usage limits

### Юридический аспект

Если через OpenClaw проходят данные клиентов или третьих лиц, это создаёт обязательства по защите персональных данных (ФЗ-152 в РФ, GDPR в ЕС).

Self-hosted снижает риск, но не снимает ответственность — данные уходят в API провайдера модели при каждом запросе к LLM.

## Чего не делать

| Действие | Почему опасно |
|---|---|
| Запускать от root | Полный доступ ко всей системе при компрометации |
| Открывать порт 18789 в интернет | CVE-2026-25253 — удалённое выполнение кода |

| Коммитить `.env` или конфиг с ключами в git | API-ключи станут публичными |
| Использовать один API-ключ на production и testing | При утечке скомпрометирован и production |
| Использовать `auth: "none"` | Любой в сети получает полный доступ |
| Привязывать к `0.0.0.0` | Открывает gateway всей сети/интернету |
| Размещать API-ключи в `skills.entries` | Неправильная структура конфига |
| Игнорировать обновления безопасности ОС | Уязвимости на уровне kernel/SSH |
| Не делать бэкапы | Потеря данных при сбое VPS |
| Не ставить usage limits у провайдера | При утечке ключа неограниченное списание |

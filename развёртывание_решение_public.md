## Решение по API-доступу

### Выбор провайдера AI API

**Провайдер**: OpenRouter (openrouter.ai)

**Причина выбора**:
- Прямой доступ к Anthropic/OpenAI может быть ограничен по геолокации
- Банковские карты из санкционных регионов не принимаются напрямую
- OpenRouter — API-агрегатор с нативной интеграцией в OpenClaw
- Доступ к 300+ моделям через один API-ключ
- Автоматический fallback при падении провайдера
- Нет прямого контракта с провайдером — ниже риск бана по геолокации

**Старт**: бесплатные модели с поддержкой tool calling (обязательно для OpenClaw)

**Риски**:
- Блокировка аккаунта OpenRouter: **низкий** (нет гео-ограничений в ToS)
- Отказ отдельных моделей по геолокации: **средний** (секция 5.7 ToS — передача страны провайдерам)
- Будущее ужесточение санкций: **неизвестно**

**Смягчение**:
- При отказе одной модели — переключение на другую
- OpenRouter — агрегатор, не зависим от одного провайдера

**Конфигурация в OpenClaw**:

```bash
openclaw onboard --auth-choice apiKey --token-provider openrouter --token "$OPENROUTER_API_KEY" --install-daemon
```

**Рекомендованные модели**:
- **Для оркестрации**: Claude Sonnet 4.5
- **Для рутины**: DeepSeek R1 (бесплатно)
- **Для кода/анализа**: GPT-4o — баланс цена/качество
- **Для драфтов**: Gemini 2.0 Flash (бесплатно)

### Ключевые архитектурные решения

1. **OpenClaw НЕ выходит в публичный интернет**: gateway и Web UI слушают только на localhost и Tailscale IP. Никаких открытых портов, кроме SSH.

2. **Telegram** — основной канал мобильного доступа. Бот подключается к Telegram Bot API outbound (OpenClaw инициирует соединение через long polling), входящих соединений не требует.

3. **Tailscale** — основной канал для CLI и Web UI. End-to-end WireGuard шифрование, zero trust.

4. **Docker sandbox mode "all"** — все операции агента выполняются внутри контейнеров, изоляция от хост-системы.

5. **OpenRouter** — единая точка доступа к AI-моделям. Gateway делает HTTPS-запросы к OpenRouter API, OpenRouter маршрутизирует к конкретным провайдерам.

### Ресурсная оценка

**2 CPU, 4 GB** достаточен для старта:
- 1-3 агента одновременно
- Базовая браузерная автоматизация
- Облачные LLM (OpenRouter)

**Сигналы для масштабирования** (4 CPU, 4-6 GB):
- RAM > 85% постоянно (5+ минут)
- CPU load average > 80% (1 min avg)
- 5-10 параллельных агентов с browser automation

## Этапы развёртывания

### Этап 0. Подготовка (до VPS)

**Что делаем:**
- Заказать VPS, Ubuntu 24.04 LTS
- Создать аккаунт Tailscale (бесплатный Personal план)
- Создать Telegram-бота через @BotFather, получить bot token
- Прописать DNS: A-запись домена → IP VPS
- Зарегистрироваться на openrouter.ai, получить API-ключ

**Результат:** VPS доступен по SSH, домен резолвится, Tailscale и Telegram-бот готовы, API-ключ OpenRouter получен.

### Этап 1. Hardening VPS (безопасность ОС)

**Что делаем:**
1. Обновить систему: `apt update && apt upgrade -y`
2. Создать рабочего пользователя (не root)
3. Настроить SSH:
   - Только key-based аутентификация (отключить пароли)
   - Отключить root login
   - Изменить порт SSH (опционально, снижает шум в логах)
4. Установить и настроить UFW:
   - `default deny incoming`, `default allow outgoing`
   - Разрешить только SSH
5. Установить fail2ban (защита от brute-force SSH)
6. Включить автообновления безопасности: `unattended-upgrades`
7. Настроить logrotate для системных логов

**Результат:** VPS принимает только SSH по ключу. Всё остальное — закрыто.

### Этап 2. Docker

**Что делаем:**
1. Установить Docker Engine (из официального репозитория Docker, не `docker.io` из Ubuntu)
2. Добавить рабочего пользователя в группу `docker`
3. Настроить Docker daemon:
   - Отключить userland-proxy
   - Ограничить логи контейнеров (max-size, max-file)
   - Не привязывать порты контейнеров к `0.0.0.0`

**Результат:** Docker готов, настроен безопасно.

### Этап 3. Node.js

**Что делаем:**
1. Установить Node.js 22 LTS через NodeSource
2. Проверить версию: `node -v`, `npm -v`

**Результат:** Runtime готов.

### Этап 4. Tailscale

**Что делаем:**
1. Установить Tailscale на VPS
2. Авторизовать VPS в Tailscale network
3. Получить Tailscale IP VPS
4. Включить MagicDNS (опционально)
5. Настроить ACL в Tailscale admin console:
   - Разрешить доступ к портам 18789, 18790 только с ваших устройств
6. Установить Tailscale на ноутбук/телефон, авторизовать
7. Проверить связность: ping VPS по Tailscale IP

**Результат:** Приватная сеть между вашими устройствами и VPS. OpenClaw будет доступен только внутри неё.

### Этап 5. Установка OpenClaw

**Что делаем:**

1. `npm install -g openclaw@<VERSION>` (фиксированная версия — не `@latest`)
2. Запустить onboarding с OpenRouter:
   ```bash
   openclaw onboard --auth-choice apiKey --token-provider openrouter --token "$OPENROUTER_API_KEY" --install-daemon
   ```
   - Пропустить настройку мессенджеров (Telegram подключим позже)
3. Проверить, что systemd user unit создан и gateway стартует
4. Тест: отправить запрос через CLI к бесплатной модели

**Результат:** OpenClaw установлен, daemon работает, подключён к OpenRouter.

### Этап 6. Безопасная конфигурация OpenClaw

**Что делаем:**

1. **Привязка к Tailscale IP**: в `~/.openclaw/openclaw.json` указать `host: "<TAILSCALE_IP>"` (не `0.0.0.0`, не публичный IP)
2. **Auth token**: сгенерировать (`openssl rand -hex 32`), указать в конфиге
3. **API-ключ OpenRouter**:
   - Создать `~/.openclaw/.env` с ключом: `OPENROUTER_API_KEY=sk-or-v1-...`
   - `chmod 600 ~/.openclaw/.env`
   - В `openclaw.json` секция `env` — ссылка `${OPENROUTER_API_KEY}` (поддерживается)
   - В `auth-profiles.json` — **реальный ключ** (подстановка переменных НЕ поддерживается)
   - Ключ в `credentials.openrouter`, НЕ в `skills.entries`
   - **Не хранить ключ в `.bak` файлах** — удалять бэкапы конфигов, содержащие ключи
4. **Права на файлы**:
   - `chmod 700 ~/.openclaw/`
   - `chmod 700 ~/.openclaw/agents/main/agent/`
   - `chmod 600 ~/.openclaw/openclaw.json`
   - `chmod 600 ~/.openclaw/.env`
   - `chmod 600 ~/.openclaw/agents/main/agent/auth-profiles.json`
5. **Docker sandbox mode "all"**: все операции внутри контейнеров
6. **Политика доступа**: `denyByDefault: true`, whitelist конкретных инструментов
7. **Запретить `group:runtime`** (exec, bash) по умолчанию
8. **Browser config**: `headless: true`, порт на localhost
9. **Spending limit на API-ключе OpenRouter**:
   - В настройках ключа на openrouter.ai → установить Credit limit
   - Ограничивает финансовые потери при утечке ключа
10. **Ротация API-ключа OpenRouter** (раз в 30 дней или при подозрении на утечку):
    1. Создать новый ключ с тем же spending limit
    2. Обновить `~/.openclaw/.env`
    3. Обновить `~/.openclaw/agents/main/agent/auth-profiles.json`
    4. `systemctl --user restart openclaw-gateway`
    5. Проверить работу агента
    6. Удалить старый ключ
11. **Ротация auth token gateway** (раз в 30 дней):
    1. `openssl rand -hex 32` — сгенерировать новый токен
    2. Обновить в конфиге gateway
    3. `systemctl --user restart openclaw-gateway`
12. **Защита от атак на цепочку поставок (supply chain)**:
    - Фиксировать версию OpenClaw при установке и обновлении (не `@latest`)
    - `npm audit --global` перед каждым обновлением
    - Снапшот VPS перед обновлением
    - Docker-образы: фиксировать теги (не `latest`), сканировать `docker scout cves`
    - Подписаться на GitHub releases OpenClaw — обновлять только после проверки changelog

**Результат:** OpenClaw изолирован, защищён auth token, ключ в .env, sandbox включён. API-ключ с лимитом расхода, ротация ключей и токенов настроена, supply chain под контролем.

### Этап 7. Telegram-бот (чат с агентом)

**Что делаем:**
1. Создать **отдельного** Telegram-бота через @BotFather (НЕ тот, что для алертов)
2. Настроить в OpenClaw канал Telegram: указать `botToken` в `channels.telegram`
3. Настроить `allowFrom` — только ваш Telegram ID (формат: `tg:ЧИСЛО`)
4. Настроить `dmPolicy: "pairing"` — бот отвечает только в личных сообщениях
5. Проверить: отправить сообщение боту → получить ответ от агента

**Два Telegram-бота:**
- **Бот алертов** — односторонние уведомления от cron-скриптов. Работает независимо от OpenClaw (curl из bash)
- **Бот чата** — двусторонний чат с AI-агентом через OpenClaw-канал. Зависит от работающего gateway

Разделение гарантирует: если OpenClaw упадёт, алерты продолжат приходить.

**Результат:** Мобильный доступ к OpenClaw через Telegram.

### Этап 8. Браузерная автоматизация

**Что делаем:**

1. Установить Chromium через Playwright (`npx playwright install chromium`) в пользовательское пространство
2. Установить системные зависимости (`sudo apt install` — libatk, libcups, libgbm, libcairo, libpango, libasound и др.)
3. Конфигурация в `openclaw.json`:
   ```json
   {
     "browser": {
       "executablePath": "<PATH_TO_PLAYWRIGHT_CHROMIUM>",
       "headless": true,
       "noSandbox": true,
       "defaultProfile": "openclaw"
     }
   }
   ```
4. Тест: `openclaw browser open https://example.com` + `openclaw browser screenshot`

⚠️ **Важно**: `noSandbox: true` обязателен для headless VPS без графической среды.

**Результат:** Браузерная автоматизация работает.

### Этап 9. Мульти-агент

**Что делаем:**

1. Переключить на платную модель с поддержкой tool calling
2. Настроить модель субагентов (`agents.defaults.subagents.model`)
3. Настроить fallback-модели
4. Профиль инструментов: `coding`, `tools.deny: ["group:runtime"]`
5. Убрать `sessions_send` из deny-списка (нужен для мульти-агента)
6. Тест: spawn субагентов параллельно

**Результат:** Мульти-агентная работа настроена.

### Этап 10. Мониторинг

**Что делаем:**

1. Создать скрипт мониторинга (проверки: RAM, CPU, swap, диск, gateway, Docker, sandbox, permissions, ports, key leak)
2. Настроить logrotate для логов мониторинга
3. Настроить `docker system prune` по расписанию (еженедельно)
4. Алерты через Telegram Bot API при WARN/CRITICAL
5. Cron `*/5 * * * *` для мониторинга
6. Cron-напоминание о ротации ключей (ежемесячно)

**Результат:** Мониторинг с активным оповещением.

### Этап 11. Бэкап

**Что делаем:**

1. GPG-ключевая пара (RSA 4096): приватный ключ — локально, публичный — на VPS
2. Скрипт бэкапа: `tar` → `gpg --encrypt` → `~/backups/`, ротация 7 дней, Telegram-уведомление
3. Бэкапить `~/.openclaw/` (конфиг, ключи, агенты, workspace) + `dpkg --get-selections` + `docker images`
4. Cron ежедневно
5. Скрипт pull-загрузки бэкапов на локальную машину через scp
6. Снапшоты VPS перед каждым обновлением OpenClaw

**Результат:** Ежедневный автоматический бэкап с шифрованием.

### Этап 12. Прокси для разблокировки моделей

**Задача:** настроить исходящий прокси для OpenClaw gateway, чтобы запросы к OpenRouter уходили с не-заблокированного IP. Разблокирует модели Anthropic и OpenAI при геоблокировке.

**Что делаем:**

1. Cloudflare WARP (SOCKS5 `127.0.0.1:40000`) — бесплатно, быстрый старт
2. Privoxy (HTTP-прокси `127.0.0.1:8118`) — обёртка над WARP SOCKS5, т.к. Node.js undici не поддерживает `socks5h://` в `HTTPS_PROXY`
3. Прокси-цепочка: OpenClaw → Privoxy (HTTP) → WARP (SOCKS5) → интернет
4. `HTTPS_PROXY` / `HTTP_PROXY` / `NO_PROXY` в systemd unit `openclaw-gateway.service`:
   ```ini
   Environment=HTTPS_PROXY=http://127.0.0.1:8118
   Environment=HTTP_PROXY=http://127.0.0.1:8118
   Environment=NO_PROXY=localhost,127.0.0.1,::1
   ```
5. Privoxy config: `forward-socks5t / 127.0.0.1:40000 .`
6. Переключить модели, протестировать

⚠️ **Важно**: OpenClaw (Node.js undici) не поддерживает `socks5h://` в переменных `HTTPS_PROXY`. Прямое подключение через SOCKS5 вызывает crash gateway (`Invalid URL protocol`). Нужен промежуточный HTTP-прокси (Privoxy).

**Fallback-стратегия**: если WARP нестабилен — модели Google и open-source в fallbacks работают без прокси. Долгосрочно — SSH SOCKS5 через VPS в другом регионе.

**Результат:** доступ ко всем платным моделям через OpenRouter.

## Важные находки

### Требование tool calling

**OpenClaw требует от модели поддержки tool/function calling.** Модели без этой возможности молча не работают — агент завершается без вывода и без ошибки в TUI.

Перед сменой модели проверять поддержку tools через curl-тест с параметром `"tools"`.

### Геоблокировка провайдеров через OpenRouter

При размещении VPS в регионах с санкционными ограничениями:
- Anthropic и OpenAI могут возвращать 403 через OpenRouter
- Google (Gemini) и open-source модели работают без ограничений
- Прокси с IP из незаблокированного региона полностью снимает блокировку
- Блокировка привязана к IP входящего запроса, не к аккаунту OpenRouter

### Побочный эффект `openclaw config set`

При каждом вызове `openclaw config set`:
- Переменная `${OPENROUTER_API_KEY}` в секции `env` заменяется на реальное значение ключа
- Права `openclaw.json` сбрасываются с `600` на `664`

Решение: использовать `jq` для изменения конфига вместо `openclaw config set`, затем `chmod 600`.

## Отложенные задачи

1. **Разведка MCP-интеграций** — исследование доступных интеграций через Model Context Protocol
2. **Ollama на отдельном сервере** — локальные модели для приватности и нулевых API-расходов (8+ GB RAM)
3. **Расширенный мониторинг** (Uptime Kuma и др.) — дашборды, графики, история
4. **Многопользовательский доступ** — `allowedUsers`, отдельные auth token, роли
5. **SSL-сертификат и nginx** — если потребуется публичный Web UI
6. **Разделение пользователей** — gateway от отдельного системного пользователя, API-ключ изолирован
7. **Обёртка над `openclaw config set`** — автоматическое восстановление ссылки на ключ и прав файла
